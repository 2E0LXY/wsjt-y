# Windows Build Pitfalls

Root-caused during the v3.0.0 first-release build cycle on MSYS2/MinGW64. Every pitfall below was hit in sequence. This document exists so future builds do not repeat that cycle.

---

## Quick checklist

Copy this into any PR or release checklist for a Windows build:

```
[ ] Hamlib not in MSYS2 pacman — build 4.5.5 from tarball
[ ] mingw-w64-x86_64-readline installed BEFORE Hamlib configure
[ ] gcc + gcc-fortran pinned to 14.2.0-3 via pacman -U --nodeps
[ ] -DWSJT_WITH_OMNIRIG=OFF passed to CMake
[ ] WSJT_NO_OMNIRIG guards all 3 OmniRig blocks in TransceiverFactory.cpp
[ ] parameter (NFFT=32768) declared in map65/libm65/decode0.f90
[ ] -fallow-argument-mismatch in General_FFLAGS (CMakeLists.txt)
[ ] FindFFTW3.cmake: NOT WIN32 condition removed from threads block
[ ] -static-libgcc -static-libstdc++ -static-libgfortran in EXE linker flags
[ ] libhamlib-utils in Linux dep install (rigctl needed by CPack)
```

---

## Pitfall 1 — Hamlib not in MSYS2

**Symptom:** `error: target not found: mingw-w64-x86_64-hamlib`

**Cause:** Hamlib is not in the MSYS2 package repository.

**Fix:** Build from source:
```bash
curl -L https://github.com/Hamlib/Hamlib/releases/download/4.5.5/hamlib-4.5.5.tar.gz | tar xz
cd hamlib-4.5.5
./configure --prefix=/mingw64 --disable-static --without-cxx-binding --disable-winradio
make -j$(nproc) && make install
```
Required extra packages: `autoconf automake libtool make`

---

## Pitfall 2 — readline API mismatch in Hamlib

**Symptom:** `error: 'rl_readline_name' undeclared`, `error: too many arguments to function 'readline'`

**Cause:** MSYS2 ships `libedit` with broken readline-compatible stubs. Hamlib's `configure` detects it as "readline support" and fails to compile.

**Fix:** Install GNU readline before configuring Hamlib:
```bash
pacman -S mingw-w64-x86_64-readline
```
Note: `--disable-readline` is not a recognised Hamlib 4.5.5 configure option — it is silently ignored.

---

## Pitfall 3 — OmniRig CMake fatal error

**Symptom:** `CMake Error: You need to install OmniRig on this computer`

**Cause:** `WSJT_WITH_OMNIRIG` defaults to `ON`. OmniRig COM server is not installed on CI runners.

**Fix:** `-DWSJT_WITH_OMNIRIG=OFF` in the cmake invocation.

---

## Pitfall 4 — OmniRig C++ compile errors

**Symptom:** `fatal error: OmniRig.h: No such file or directory` (after pitfall 3 fix)

**Cause:** `OmniRig.h` is generated by qaxcontainer only when OmniRig is enabled. `TransceiverFactory.cpp` had three `#if defined(WIN32)` guards that did not check `WSJT_WITH_OMNIRIG`.

**Fix:** Three-part source change (already applied in this repo):
1. CMakeLists.txt: add `WSJT_NO_OMNIRIG` compile definition to wsjt_qt target when disabled
2. TransceiverFactory.cpp: all three guards → `#if defined(WIN32) && !defined(WSJT_NO_OMNIRIG)`

## Pitfall 5a — GCC 15/16 `.tls_common` regression

**Symptom:** `Error: .tls_common directive is not supported` during Fortran compile

**Cause:** GCC 15 changed the default TLS model for MinGW targets. Fortran `SAVE` variables with OpenMP emit `.tls_common` directives that GCC 15+/MinGW assembler rejects.

**Fix:** Pin compiler binaries to 14.2.0-3:
```bash
BASE=https://repo.msys2.org/mingw/mingw64
pacman -U --noconfirm --nodeps \
  "${BASE}/mingw-w64-x86_64-gcc-14.2.0-3-any.pkg.tar.zst" \
  "${BASE}/mingw-w64-x86_64-gcc-fortran-14.2.0-3-any.pkg.tar.zst"
```

## Pitfall 5b — `__emutls_v_ZSt11__once_call` not found on install

**Symptom:** `wsjtx.exe — Entry Point Not Found: __emutls_v_ZSt11__once_call`

**Cause:** GCC 14's `libstdc++` uses emulated TLS (`__emutls_v_*` symbols) for `std::call_once`. GCC 16's `libstdc++` uses a different approach and does not export this symbol. If the installed `libstdc++-6.dll` is from GCC 16, GCC 14-compiled code cannot find its symbol.

---

## Pitfall 5c — `_ZSt15__get_once_callv` not found in `libicuuc78.dll`

**Symptom:** `wsjtx.exe — Entry Point Not Found: _ZSt15__get_once_callv could not be located in libicuuc78.dll`

**Cause:** The inverse of 5b. `libicuuc78.dll` (Qt's ICU library, compiled with GCC 16) imports `_ZSt15__get_once_callv` (`std::__get_once_call()`) from `libstdc++-6.dll`. If we replaced the bundled `libstdc++-6.dll` with GCC 14's version (5b fix), ICU can no longer find this symbol.

**The catch-22:**
| libstdc++ version | wsjtx.exe | libicuuc78.dll |
|------------------|-----------|----------------|
| GCC 14 | ✓ `__emutls_v_ZSt11__once_call` | ✗ missing `_ZSt15__get_once_callv` |
| GCC 16 | ✗ missing `__emutls_v_ZSt11__once_call` | ✓ `_ZSt15__get_once_callv` |

**Definitive fix:** Statically link the GCC runtime into every executable:
```cmake
if (WIN32 AND MINGW)
  set (CMAKE_EXE_LINKER_FLAGS
       "${CMAKE_EXE_LINKER_FLAGS} -static-libgcc -static-libstdc++ -static-libgfortran")
endif ()
```
Each `.exe` carries its own copy of GCC 14's libstdc++. Qt/ICU/Boost continue using GCC 16's shared runtime. No conflict.

**Effect:** Each exe is ~3–5 MB larger. `libstdc++-6.dll` and `libgcc_s_seh-1.dll` are no longer needed by the executables.

## Pitfall 6 — `NFFT` undeclared in decode0.f90

**Symptom:** `Error: Variable 'nfft' cannot appear in the expression at (1)`

**Cause:** `decode0.f90` uses `NFFT` as an explicit array-shape bound without declaring it. Older GCC accepted it as an implicit external reference; GFortran 14 requires explicit declaration.

**Fix:** Added `parameter (NFFT=32768)` to `decode0.f90` (already applied in this repo).

---

## Pitfall 7 — MAP65 Fortran type-punning

**Symptom:** `Error: Type mismatch in argument 'dd' at (1); passed INTEGER(1) to REAL(4)`

**Cause:** MAP65 uses a shared byte buffer passed to subroutines expecting `REAL(4)`. Pre-Fortran-90 pattern. GFortran 14 treats it as an error.

**Fix:** `-fallow-argument-mismatch` in `General_FFLAGS` in CMakeLists.txt (already applied).

---

## Pitfall 8 — `fftwf_*_threads` undefined at link

**Symptom:** `undefined reference to 'fftwf_init_threads'`

**Cause:** `FindFFTW3.cmake` had `if (NOT WIN32 AND _use_threads)` — excluded the threads library on all Windows targets. MSYS2 FFTW3 is modular; `libfftw3f_threads.dll.a` is separate.

**Fix:** Removed `NOT WIN32` condition (already applied in this repo).

---

## Why not JTSDK?

The official WSJT-X Windows releases use JTSDK (WSJT Software Development Kit), a custom interactive installer. JTSDK cannot be used in CI because it requires an interactive Windows desktop and assumes specific paths not compatible with GitHub Actions runners.

MSYS2 provides equivalent tooling via `pacman`. The pitfalls above are the one-time mapping cost.
